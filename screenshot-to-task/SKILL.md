# Screenshot to Task

将产品截图、UI 稿和 PRD 交互说明转化为可执行的前端任务 Markdown。要求**全方位理解截图和产品意图**，产出写入 `task/{中文任务名称}.md`，截图归档至 `task/screenshot/{中文任务名称}/`。

核心原则：

- **产品意图优先**：先弄清做什么、怎么做、从哪到哪，再拆任务
- **仅前端范围**：不写后端逻辑、不写部署联调顺序
- **不确定必须问**：用户确认后才继续
- **截图仅为通用输入**，不绑定任何特定业务域

## 工作流

### Phase 0 — 截图归档

在分析开始前，将用户提供的每张截图保存到：

```text
{用户项目根目录}/task/screenshot/{中文任务名称}/
├── 01-{简要描述}.png
├── 02-{简要描述}.png
└── ...
```

规则：

1. `task/screenshot/{中文任务名称}/` 目录不存在时自动创建
2. 文件名格式：`{两位序号}-{简要描述}.{原扩展名}`
3. 任务名称尚未确定时先用 `待命名`，Phase 5 确定后同步重命名
4. 用户改名时，截图目录与 `.md` 文件一并重命名

### Phase 1 — 读图与产品意图理解

对每张截图分类并提取信息。详细 checklist 见 [rules/extraction.md](rules/extraction.md)。

**必须回答的五个核心问题**（读图阶段逐项落实）：

| 问题 | 说明 |
|------|------|
| **做什么** | 本功能的产品目标是什么？用户完成什么业务动作？ |
| **怎么做** | 前端通过哪些页面、组件、交互来实现？ |
| **从哪里到哪里** | 完整用户路径：入口 → 中间页 → 终点 |
| **新建什么页面** | 哪些是全新页面/路由？哪些是改造已有页面？ |
| **点什么出什么** | 每个关键按钮的触发动作：跳转 / 弹窗 / 抽屉 / 提交 / 状态变更 |

同时提取：

1. **业务线**：从菜单路径、产品名称、平台 Logo 等推断（如「会员营销」「补贴管理」「商户小程序」）
2. **端**：小程序 / H5 / 管理后台
3. 页面标题、菜单路径、字段、按钮、状态、分页、抽屉/弹窗
4. 图间跳转关系、流程编号、层级关系

**信息源优先级**（冲突时）：PRD 文字 > UI 标注 > UI 视觉推断

#### 用户确认门禁（硬约束）

遇到不清楚的交互、字段、流转或任何需猜测的内容，**必须暂停**，列出问题等用户确认或给出修改方案后，才能进入 Phase 2。格式见 [rules/extraction.md](rules/extraction.md)。

### Phase 2 — 交互建模

1. **用户路径图**：从入口到终点的完整链路（文字或 mermaid）
2. **点击操作表**：`{页面} → 点击「{按钮}」→ {结果}` 逐条列出
3. **页面清单**：新建 / 改造 / 复用，标注每页职责
4. 状态机与操作矩阵（如有）
5. 前端业务规则：校验、Toast 文案、计算公式（展示用）、文件限制等

**排除**：后端接口实现、定时任务、数据库、部署顺序 — 这些不写入任务，若截图涉及则记入「范围外」。

### Phase 3 — 前端任务拆解

按固定结构拆解。模板见 [rules/task-template.md](rules/task-template.md)。

章节顺序（`## 0` 必须是整体任务预览；`## 11` 待补充项必须是最后一章）：

0. **整体任务预览**（**必填，文档第一个章节**）
1. 路由与菜单
2. API 与类型（仅前端消费层：接口调用、TS 类型，不写后端实现）
3. 常量与枚举
4. 页面（按页面分节，标注新建/改造）
5. 组件（弹窗、抽屉等）
6. 交互与状态流转（详细点击操作表）
7. 校验与错误处理
8. 权限
9. 验证清单
10. 范围外（后端逻辑等待排除项放这里）
11. 待补充项（**不可省略**）

每条任务使用 `- [ ]` checkbox，标注截图来源：

```markdown
- 来源：[01-列表页](screenshot/会员积分兑换/01-列表页.png)
```

### Phase 4 — 代码库映射（可选）

若用户项目存在 `package.json` 和 `src/`，扫描项目结构将任务细化到文件路径。优先读取 `src/router/`、`src/views/`、`src/api/`、`src/constants/`、`src/components/`。

- **有代码库**：细化到文件路径和代码片段
- **无代码库**：`{待映射}` 占位，禁止编造路径

### Phase 5 — 命名与落盘

产出路径：

```text
{用户项目根目录}/task/{中文任务名称}.md
{用户项目根目录}/task/screenshot/{中文任务名称}/*
```

任务名称：2～12 个汉字，取页面标题或菜单末级名称，非法字符移除，重名追加序号。

生成后提醒用户确认/修改任务名称，并提醒填写「待补充项」。

## 规范加载

| 文件 | 用途 |
| ---- | ---- |
| [rules/task-template.md](rules/task-template.md) | 输出模板，含整体任务预览格式 |
| [rules/extraction.md](rules/extraction.md) | 产品意图提取 checklist |
| [rules/standards.md](rules/standards.md) | 质量门槛 |

参考示例：[examples/example-task-format.md](examples/example-task-format.md)

## 质量标准

- **§0 整体任务预览**必填，含业务线、五核心问题、页面清单、核心用户路径
- 每条任务可追溯（截图链接）
- Toast / 文案 / 公式使用 PRD 原文
- **禁止**写入后端实现任务和部署联调顺序
- 必须有验证清单、范围外、待补充项
- 截图归档并在文档中引用

## 与 OpenSpec 的关系

产出独立 `task/*.md`，不自动写入 OpenSpec。用户可手动复制到 `openspec/changes/*/tasks.md` 后运行 `/opsx:apply`。

## 最终摘要

完成后输出：

- 文件路径（`task/*.md` + `task/screenshot/{任务名}/`）
- 业务线、任务名称及改名提醒
- 新建页面数、核心用户路径条数
- 读图阶段已确认的问题
- 提醒填写「待补充项」
