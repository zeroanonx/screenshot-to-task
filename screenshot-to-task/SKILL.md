# Screenshot to Task

将产品截图、UI 稿和 PRD 交互说明转化为可执行的前端任务 Markdown。

核心原则：

- **有任何不理解的地方，必须询问用户，由用户确认后才能继续**
- **落地文档是最后一步**：先完成全部确认和补充收集，再写入 `task/*.md`
- **§0 整体任务预览尽可能详细**
- **路由须先问用户**；**落地前须问用户是否还有补充项**
- **仅前端范围**：不含 API 章节、验证清单、后端逻辑

## 工作流总览

```text
Phase 0   截图归档（可先保存，便于引用）
Phase 1   读图 + 产品意图理解
            └─ 有任何不理解 → 暂停询问 → 用户确认
Phase 2   交互建模
            └─ 建模中仍有疑问 → 暂停询问 → 用户确认
Phase 2.5 路由确认（询问用户）
Phase 3   任务拆解（仅在内存/对话中准备，不写文件）
Phase 4   代码库映射（可选，仍在对话中准备）
Phase 5   落地前确认 ← 必须执行，用户确认后才进入 Phase 6
Phase 6   落盘写入 task/*.md ← 最后一步
```

**硬约束：Phase 6 之前禁止写入 `task/{中文任务名称}.md`。**

---

### Phase 0 — 截图归档

```text
{用户项目根目录}/task/screenshot/{中文任务名称}/
├── 01-{简要描述}.png
└── ...
```

任务名未定时先用 `待命名`，Phase 6 确定后同步重命名。

---

### Phase 1 — 读图与产品意图理解

见 [rules/extraction.md](rules/extraction.md)。落实五问，识别业务线和端。

#### 用户确认门禁（全程适用）

**任何时候**出现不理解、不确定、有歧义、需猜测的内容，**必须立即暂停**，不得继续后续阶段，不得自行推断。

包括但不限于：

- 交互逻辑、跳转目标、弹窗/抽屉行为
- 字段含义、数据来源、展示规则
- 业务线、产品意图、页面归属（新建/改造）
- PRD 与 UI 冲突
- 状态流转、按钮显示条件
- 路由方案（见 Phase 2.5）
- 任何截图看不清、理解不到位的地方

暂停时向用户列出：

```text
## 需您确认

### 问题 1：{不清楚的点}
- 我的理解：{如有}
- 涉及截图：[01-xxx](screenshot/{任务名}/01-xxx.png)
- 可选方案：A. … / B. …

请确认正确理解，或给出修改意见。确认后我继续。
```

用户确认后，将结论记入任务依据，**然后才能继续**。

---

### Phase 2 — 交互建模

为详尽 §0 准备素材：用户路径、全量点击操作表、页面清单、字段表、状态矩阵、业务规则、权限。

建模过程中新产生的疑问 → 回到用户确认门禁，**暂停询问**。

---

### Phase 2.5 — 路由确认

写入 §1 前**必须询问用户**：

1. 是否新建路由？
   - **新建** → 严格获取 path、name、菜单位置，禁止自行推断
   - **不新建** → 获取源 path、name、页面文件路径、改造方式

用户未明确回复前，不得准备 §1 内容。

---

### Phase 3 — 前端任务拆解（准备阶段，不写文件）

在对话中按模板准备全部章节内容。模板见 [rules/task-template.md](rules/task-template.md)。

章节：§0～§9（§0 尽可能详细；§9 待补充项先留空占位）。

**此阶段只准备内容，不写入磁盘。**

---

### Phase 4 — 代码库映射（可选，不写文件）

有 `package.json` 和 `src/` 时，在对话中细化文件路径。路由仍以用户确认为准。

---

### Phase 5 — 落地前确认（硬约束，不可跳过）

**在写入任何 `task/*.md` 之前，必须向用户呈现确认摘要并等待回复。**

#### 5.1 呈现理解摘要

在对话中输出完整摘要，至少包含：

- 任务名称（自动推断的中文名）
- 业务线、端
- 做什么 / 怎么做（精简版）
- 页面全景（新建/改造）
- 核心用户路径
- 路由方案（用户已确认的结果）
- 读图阶段所有已确认问题的结论

#### 5.2 询问用户（固定话术）

```text
以上是我对截图和需求的理解，请您确认：

1. 理解是否准确？如有偏差请指出，我会修正后再落地。
2. 是否还有需要补充的内容？（如遗漏的交互、特殊逻辑、技术约束等）

确认无误且补充完毕后，我将生成 task/{任务名称}.md。
```

使用 `AskQuestion` 或等待用户文字回复。**必须收到用户明确确认后，才能进入 Phase 6。**

用户补充的内容：
- 纳入 §0 或对应章节
- 写入 §9 待补充项（用户表示「后续再补」时）或直接融入任务

用户指出偏差时：
- 修正理解 → 若涉及新疑问，再次确认 → 重新呈现摘要 → 再等用户确认

---

### Phase 6 — 落盘（最后一步）

**仅在 Phase 5 用户确认后执行。**

写入：

```text
{项目根}/task/{中文任务名称}.md
{项目根}/task/screenshot/{中文任务名称}/*
```

落盘后输出：

```text
已生成 task/{中文任务名称}.md
截图目录：task/screenshot/{中文任务名称}/
任务名称由截图推断为「{中文任务名称}」，如需修改请告诉我。
§9 待补充项已预留，可随时补充。
```

## 质量标准

- 全程：有任何不理解必须问，用户确认后才继续
- Phase 5 落地前确认不可跳过
- Phase 6 是唯一步骤可写入 `task/*.md`
- §0 尽可能详细；§1 路由来自用户确认
- 禁止 API 章节、验证清单章节

## 最终摘要（Phase 6 完成后）

- 文件路径、业务线、任务名称及改名提醒
- 用户补充项是否已纳入
- §9 待补充项说明
