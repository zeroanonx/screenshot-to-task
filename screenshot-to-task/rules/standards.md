# 任务生成质量标准

> 与 [SKILL.md](../SKILL.md) 配套。

## 一、必达项

| # | 规则 | 不满足时 |
|---|------|---------|
| 1 | **有任何不理解必须询问用户，用户确认后才继续** | 阻塞交付 |
| 2 | **落地前必须询问用户确认理解 + 是否还有补充项** | 阻塞交付 |
| 3 | **`task/*.md` 仅在用户确认后写入（Phase 6）** | 阻塞交付 |
| 4 | 产出 `task/{中文任务名称}.md` + 截图归档 | 阻塞交付 |
| 5 | §0 详尽必填；§1 路由来自用户确认 | 阻塞交付 |
| 6 | §8 范围外、§9 待补充项（最后一章） | 阻塞交付 |
| 7 | 禁止 API 章节、验证清单章节 | 阻塞交付 |
| 8 | 含代码任务须标注参照文件，关键函数/变量须要求 JSDoc | 阻塞交付 |

## 二点五、代码风格与 JSDoc

任务涉及代码实现时（Phase 4 有代码库，或 §3～§5 页面/组件任务）：

1. **必须先读项目现有代码**，找同模块相似文件作参照
2. 任务中注明「参照 `路径`」，目录/命名/写法与项目一致
3. **禁止**引入项目未使用的技术栈或目录结构
4. 代码示例或实现要求中，**关键函数、复杂变量、props/emits 须有标准 JSDoc**：

```ts
/**
 * @description 函数用途
 * @param {Type} name - 参数说明
 * @returns {ReturnType}
 */
```

5. 复杂分支逻辑须加意图注释

详见 [coding-conventions.md](coding-conventions.md)。若项目已有注释风格，以项目为准。

## 三、用户确认门禁（全程）

**任何阶段**，出现以下情况必须**立即暂停**、询问用户、**等用户确认后**才能继续：

- 交互、跳转、弹窗/抽屉行为不清楚
- 字段含义、展示规则、数据来源不确定
- 业务线、产品意图、页面归属有歧义
- PRD 与 UI 冲突
- 状态流转、按钮条件无法判断
- 路由 path/name 未获用户明确提供
- 截图看不清、任何需要猜测的内容

**禁止**：自行推断后静默继续、跳过确认直接写文件。

## 四、落地前确认（Phase 5，不可跳过）

写入 `task/*.md` **之前**：

1. **呈现理解摘要**
2. **必须使用 `AskQuestion` 弹出选择框**，选项固定为：
   - `理解准确，直接生成文档`（Recommended）
   - `理解有偏差，需要修正`
   - `需要补充内容`
3. **分支处理**：
   - 选「理解准确」→ Phase 6 落盘
   - 选「需要修正」→ 请用户输入偏差 → 修正 → 重新弹选择框
   - 选「需要补充」→ **请用户输入补充内容** → 纳入文档 → 重新弹选择框
4. 循环直到用户选择「理解准确，直接生成文档」

**禁止**：仅用文字询问、跳过选择框、未选「理解准确」就落盘、选「补充」后不等待输入。

## 五、落盘顺序

```text
截图归档（Phase 0，可先执行）
  → 读图/建模/路由确认（Phase 1～2.5，全程可暂停询问）
  → 对话中准备任务内容（Phase 3～4，不写文件）
  → 落地前确认（Phase 5，等用户确认 + 收集补充项）
  → 写入 task/*.md（Phase 6，最后一步）
```

## 五、§0 详细度要求

见 [task-template.md](task-template.md)。禁止敷衍。

## 六、§1 路由询问

新建 → 严格获取 path、name、菜单位置；不新建 → 获取源 path、name、页面路径。禁止自行推断。

## 七、禁止的章节

API 与类型、验证清单、部署与联调顺序。

## 八、禁止事项

- 禁止跳过用户确认直接落盘
- 禁止跳过落地前选择框确认
- 禁止在 Phase 5 之前写入 `task/*.md`
- 禁止自行编造路由
- 禁止对不理解的内容自行猜测
- 禁止代码任务不参照项目风格、关键逻辑无 JSDoc
